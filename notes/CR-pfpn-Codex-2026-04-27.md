IMPORTANT: [crates/ht-render/src/renderer.rs:857] The valid-env override log still reports the old pixel-budget-only value as "auto-pick would have been". After this patch, the auto default is class-aware whenever the adapter is classifiable. On the defended dGPU case, `HASH_THING_RENDER_SCALE=0.75` at 2560x1440 / 256^3 would log an auto pick near 0.736 even though the real no-env auto pick is now 1.0. That directly undercuts the new diagnostics and matches the trident pack concern that existing env/CLI logs must stop using stale pixel-only "auto" language. Use the same class-aware decision helper for this diagnostic, or phrase it explicitly as `pixel_pick`.

IMPORTANT: [crates/ht-render/src/renderer.rs:614] `DeviceType::DiscreteGpu => 1.0` is broader than the stated spec anchor "RTX 3060+ -> 1.0". This also leaves the trident-plan concern about old/mobile discrete GPUs unaddressed: GT 1030, MX-series, low-power laptop dGPUs, and GTX 1650-class cards all classify as `DiscreteDgpu` and start native. If the product decision is really "all discrete GPUs get 1.0", the spec/docs should say that. If the intended threshold is RTX-3060-class, this classifier needs at least a conservative low-end-name fallback before merge.
